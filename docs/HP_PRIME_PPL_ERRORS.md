# HP_PRIME_PPL_ERRORS.md
## Типовые ошибки в файлах HP Prime PPL и как их не повторять

**Статус:** Постоянный справочник — пополнять при обнаружении новых паттернов.

---

## 1. Служебная первая строка `.hpprgm` вставлена как обычный код

### Что было
В файлах для HP Prime сверху была строка вида:
```
editor_shell;
test_editor_shell_basic;
renderer;
search_engine;
```

### Почему это ошибка
Эта строка нужна как заголовок контейнера `.hpprgm`, но **не является PPL-кодом** для ручной вставки в редактор HP Prime. Если вставить её вручную:
- ломает разбор файла;
- показывает ошибку не на первой строке, а позже;
- например уже на `EXPORT ...`, хотя реальная причина именно в первой строке.

### Как правильно
- Если используешь HP Connectivity Kit / Import — строка допустима.
- Если копируешь код вручную в редактор программы HP Prime — **удаляй первую строку всегда**.

### Правило
> Никогда не вставлять строку `program_name;` вручную в редактор HP Prime.

## 2. `EXIT` внутри циклов и блоков вызывает синтаксические ошибки

### Что было
В коде встречались конструкции типа:
```ppl
IF bx < R_X_LEFT THEN EXIT; END;
```
или:
```ppl
EXIT;  // stop wrap pass at the start line
```

### Почему это ошибка
В HP Prime PPL `EXIT` в таких местах:
- либо не поддерживается так, как ожидалось;
- либо парсится нестабильно;
- либо даёт синтаксическую ошибку в зависимости от контекста.

Особенно проблематичны:
- `EXIT` внутри `FOR`
- `EXIT` внутри вложенного `IF` в цикле
- `EXIT` как early-break внутри search / renderer logic

### Как правильно
Переписывать логику **без `EXIT`**, через:
- ограничение диапазона цикла;
- флаг завершения;
- дополнительное условие `IF ... THEN ... END`;
- проверку границы до выполнения тела.

### Пример
**Было:**
```ppl
IF bx < R_X_LEFT THEN EXIT; END;
// render
```

**Стало:**
```ppl
IF bx >= R_X_LEFT THEN
  // render
END;
```

**Для поиска (wrap pass). Было:**
```ppl
IF ln == start_line THEN
  ...
  EXIT;
END;
```

**Стало:** цикл идёт только до `start_line`; на `start_line` применяется отдельное условие; `EXIT` не нужен.

### Правило
> Не использовать `EXIT` как универсальный break в HP Prime PPL.
> Сразу проектировать цикл без необходимости `EXIT` — **"bounded loops"**, не **"break-style loops"**.

## 3. Внешний тестовый файл вызывал функции, которые HP Prime не видел

### Что было
В тестовых файлах вызывались функции из другого модуля:
```ppl
editor_shell();
editor_shell_from(doc);
ES__Run();
```

### Почему это ломалось
Хотя функции экспортированы в основном модуле, HP Prime:
- не всегда корректно разрешает внешние зависимости между программами;
- может ругаться на вызов корректной функции, если в момент компиляции тестовый файл её «не видит»;
- особенно если модуль загружен в неправильном порядке.

Причины на практике:
- основной модуль не загружен;
- загружен старый вариант;
- имя программы/экспорта не совпало;
- HP Prime компилирует текущий файл до разрешения зависимостей.

### Как правильно
- Сначала загружать и сохранять основной модуль, потом компилировать тест.
- Держать demo/blank launchers в том же модуле, что и основной runtime.
- Для критичных launchers не плодить лишние внешние обёртки.

### Правило
> Чем меньше межмодульных вызовов в test harness, тем надёжнее компиляция на HP Prime.

## 4. HP Prime показывает ошибку не на строке-причине, а «рядом»

### Что было
HP Prime показывал ошибку, например, на:
```
EXPORT editor_shell()
EXPORT test_renderer_basic()
EXPORT test_editor_shell_basic()
```

### Почему это вводило в заблуждение
Синтаксическая ошибка часто показывается:
- **не на корневой проблеме**;
- а на строке, где парсер «окончательно потерял структуру».

Реальная причина могла быть:
- выше по файлу;
- в предыдущем блоке;
- в лишней первой строке контейнера;
- в неподдерживаемом `EXIT`;
- в незакрытом или плохо распарсенном блоке.

### Как правильно
Если ошибка показывает на `EXPORT ...`, сначала проверять:
- нет ли сверху строки `program_name;`
- нет ли выше `EXIT`
- не сломан ли предыдущий `BEGIN/END`
- нет ли неподдерживаемой конструкции в предыдущем блоке

### Правило
> Если HP Prime ругается на `EXPORT`, **сначала ищи проблему выше по файлу**.

## 5. Смешивались два сценария использования файлов: Import vs manual paste

### Что было
Один и тот же текст использовали одинаково: и для импорта `.hpprgm`, и для ручной вставки.

### Почему это ошибка
У сценариев разные требования:

| Сценарий | Требование |
|---|---|
| Import через Connectivity Kit | Допускает контейнерный формат с `program_name;` в первой строке |
| Ручная вставка в редактор | Требует чистый PPL-код **без** этой строки |

### Правило
> Для HP Prime всегда различать два режима: **import-file** и **paste-into-editor**. Не смешивать.

## 6. Предположения о доступности функций без проверки загруженного API

### Что было
Предполагалось, что можно вызывать `editor_shell()`, `ES__Run()` из внешнего теста, не проверяя загруженную версию модуля.

### Почему это ошибка
На HP Prime важно не только то, что функция есть в исходнике, но и то:
- загружена ли эта версия;
- сохранена ли она;
- совпадает ли имя программы и экспорт с ожидаемым.

### Как правильно
- Проверить фактический экспорт.
- Запускать сначала основной entrypoint напрямую из Home.
- Только потом писать launcher/test harness вокруг него.

### Правило
> Сначала убедиться, что основной модуль реально запускается сам по себе. Только потом строить внешний тестовый слой.

## 7. В тест-файлах были лишние зависимости вместо вызова основного сценария напрямую

### Что было
Для shell и renderer создавались отдельные тестовые программы, которые просто переадресовывали вызов.

### Почему это плохо
На HP Prime каждая дополнительная внешняя программа:
- добавляет точку отказа;
- требует разрешения межмодульной зависимости;
- повышает риск «undefined / syntax error on call».

### Как правильно
Если задача теста — просто запустить сценарий:
- лучше добавлять `*_demo()` или `*_blank()` прямо в основной модуль;
- отдельный test harness — только когда он реально даёт дополнительную проверочную ценность.

### Правило
> Launcher-функции лучше держать рядом с runtime-модулем, а не выносить без необходимости.

## 8. Search / renderer logic: ранний выход через `EXIT` ломает компиляцию

*(см. также пункт 2 — общий случай)*

### Что было
В поиске и рендерере ранний выход пытались делать через `EXIT`.

### Как правильно
Проектировать логику так, чтобы:
- цикл сразу имел правильные границы;
- wrap-pass был ограничен диапазоном (`FOR ln FROM 1 TO stop_line`);
- clipping был условным, а не break-based.

### Правило
> В HP Prime PPL использовать **bounded loops**, не **break-style loops**.

## Краткий чеклист перед загрузкой/вставкой нового файла

### Если вставляю вручную
- [ ] Удалена первая строка `program_name;`
- [ ] Нет `EXIT` в проблемных местах
- [ ] `LOCAL` объявлены корректно (все в начале функции)
- [ ] Предыдущий блок `BEGIN/END` не сломан
- [ ] Основной модуль уже загружен и сохранён

### Если делаю test harness
- [ ] Не вызываю лишние внутренние функции из другого файла без необходимости
- [ ] Сначала проверил основной entrypoint напрямую
- [ ] Зависимости загружены в правильном порядке

### Если ошибка показывает на `EXPORT`
- [ ] Проверяю строку выше
- [ ] Проверяю первую строку файла
- [ ] Проверяю предыдущий цикл/IF
- [ ] Проверяю наличие `EXIT`

---

## Главное правило

Для HP Prime нужно всегда различать:
- формат файла для импорта vs чистый код для ручной вставки;
- что реально экспортировано и уже загружено в память;
- что парсер HP Prime показывает как место ошибки, и что является **реальной причиной** (часто — выше по файлу).

> Иначе можно долго исправлять «не ту строку», хотя реальная проблема находится выше.
